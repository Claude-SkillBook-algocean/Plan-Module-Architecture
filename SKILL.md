# Hierarchical SRP Module Architecture Design

계층적 단일책임 원칙(SRP)에 기반한 모듈 설계 스킬. 
개발 전 철저한 설계를 수행하여, 설계서만 보고 바로 구현에 들어갈 수 있는 수준의 상세한 HTML 설계서를 생성한다.

## 핵심 철학

> **"하나의 모듈은 하나의 책임만, OUT은 반드시 1개, 메인 파이프라인은 직선 1줄"**

## 워크플로우 (반드시 이 순서를 따른다)

### Phase 1: 기술스택 협의 (Tech Stack Discussion)

사용자에게 프로젝트 목표를 파악한 후, 기술스택 후보를 제안하고 **반드시 사용자 확인을 받은 후** 다음 단계로 진행한다.

질문해야 할 것:
1. 프로젝트의 핵심 목적과 도메인
2. 배포 환경 (로컬/클라우드/하이브리드)
3. 성능 요구사항 (실시간/배치/혼합)
4. 사용자의 기술 선호도 및 경험
5. 예산 및 인프라 제약사항

기술스택 확정 후 표로 정리하여 사용자에게 보여주고 승인받는다.

### Phase 2: 모듈 계층 설계 (Module Hierarchy Design)

**반드시 `references/module-design-rules.md`를 읽고** 다음 원칙을 적용한다.

1. **프로젝트를 기능 단위(Feature)로 분해**: F1, F2, F3... + C0(공통모듈)
2. **각 Feature를 최소 단위 모듈로 분해**: F1.1, F1.2, F1.3...
3. **각 모듈이 SRP를 만족하는지 검증**: "이 모듈의 변경 이유가 2가지 이상이면 분할"
4. **메인 파이프라인 도출**: 각 Feature의 핵심 흐름을 직선 1줄로 구성
5. **모듈 간 의존성 맵 작성**: 어떤 모듈이 어떤 공통모듈/다른 Feature 모듈을 사용하는지

### Phase 3: 상세 IN/OUT 명세 (Detailed I/O Specification)

모든 모듈에 대해 다음을 **빠짐없이** 작성한다:

```
[모듈 ID] 모듈명 — 한 줄 설명
├─ IN (메인): 파라미터명 (타입) — 출처모듈ID, 설명
├─ IN (보조1): 파라미터명 (타입) — 출처모듈ID, 설명  
├─ IN (보조N): ...
├─ OUT: 반환타입명 (필드1: 타입, 필드2: 타입, ...)
├─ 내부로직: 핵심 처리 과정 설명 (어떤 알고리즘/패턴)
├─ 사용 공통모듈: C0.x, C0.y
└─ 제약사항/주의점: ...
```

### Phase 4: HTML 설계서 생성

**반드시 `references/html-template-guide.md`를 읽고** HTML 설계서를 생성한다.

HTML 설계서에 반드시 포함되어야 하는 섹션:
1. **프로젝트 헤더**: 프로젝트명, 버전, 설명, 생성일
2. **기술스택 테이블**: 영역별 기술/역할/배포환경
3. **프로젝트 폴더 구조**: 트리 형태 (모듈ID 주석 포함)
4. **전체 메인 파이프라인**: 직선형 흐름도
5. **공통 모듈 (C0)**: 각 모듈별 상세 IN/OUT + 내부 설계
6. **기능 모듈 (F1~FN)**: 각 Feature별 카드
   - Feature 파이프라인 (해당 Feature 내부의 직선 흐름)
   - 하위 모듈 상세 (접기/펴기 토글)
   - 각 모듈의 IN/OUT 완전 명세
7. **의존성 매트릭스**: 모듈 간 참조 관계
8. **체크리스트**: 구현 순서 및 진행 추적

## 모듈 설계 핵심 규칙 (Quick Reference)

상세 규칙은 `references/module-design-rules.md` 참조.

| 규칙 | 설명 |
|------|------|
| **OUT = 1** | 하나의 모듈은 반드시 1개의 OUT만 가진다 |
| **IN ≥ 1** | IN은 1개 이상, 메인 IN과 보조 IN으로 구분 |
| **메인 파이프라인 = 직선** | 분기 없는 1줄 파이프라인 |
| **SRP 엄수** | 변경 이유가 2가지 이상이면 반드시 분할 |
| **최하위 = 원자적** | 최하위 모듈은 단일 기능 (크롤링, 파싱, 계산 등) |
| **명세 = 코드 수준** | IN/OUT에 함수명, 변수명, 타입 모두 기재 |
| **공통모듈 독립** | C0은 Feature와 분리, DI로 주입 |

## HTML 설계서 품질 기준

상세 가이드는 `references/html-template-guide.md` 참조.

- 다크 테마 기반 (눈의 피로 최소화)
- JetBrains Mono (코드) + Noto Sans KR (본문) 폰트
- 모듈별 색상 구분 (Feature마다 고유 컬러)
- 접기/펼치기 토글 (전체 펼치기/접기 버튼 포함)
- IN은 녹색 계열, OUT은 주황색 계열, API는 시안 계열로 색상 코딩
- 메인 파이프라인은 화살표(→) 연결 시각화
- 모듈 내부의 서브 모듈 관계도 시각적으로 표현
- 모든 타입/변수명은 모노스페이스 폰트로 표시
- 반응형 디자인 (모바일 확인 가능)

## 대규모 프로젝트 처리

모듈이 30개 이상이 되는 대규모 프로젝트의 경우:

1. Phase 2까지 완료 후 전체 로드맵을 사용자에게 보여준다
2. C0(공통모듈) → F1 → F2 → ... 순서로 Feature 단위로 나누어 작성
3. 각 Feature 작성 완료 시 "다음 Feature를 이어서 작성할까요?" 확인
4. 마지막에 전체를 하나의 HTML로 통합
5. 작성 진행 상황을 `[현재: FN 작성 완료 / 다음: FM 예정]` 형태로 표시
